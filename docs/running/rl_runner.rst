RLRunner: Single‑Match Execution
================================

The ``RLRunner`` module is responsible for executing **a single complete
tactical match** using a trained reinforcement learning policy and
producing a **deterministic, engine‑native replay**.

Its purpose is to provide a clean and reproducible bridge between:

- a trained PPO policy
- the deterministic tactical engine
- the replay and visualization pipeline

The runner is intentionally minimal and narrowly scoped.

Scope and responsibilities
--------------------------

The ``RLRunner`` is responsible for:

- Loading a trained PPO model **for inference only**
- Executing exactly **one full match**
- Forwarding agent actions to the environment
- Recording all engine‑generated state transitions
- Persisting a replay file to disk

The ``RLRunner`` explicitly does **not**:

- Train agents
- Execute batches of matches
- Control or configure the opponent
- Compute statistics or diagnostics
- Evaluate performance or outcomes
- Modify engine rules or reward signals

This strict scope prevents accidental coupling with
training or analysis logic.

Single‑episode semantics
------------------------

Each ``RLRunner`` invocation corresponds to **one and only one match**.

This design decision is deliberate and ensures that:

- Replays have a clear semantic meaning
- Episode boundaries are explicit
- Deterministic reproduction is straightforward
- Execution logic remains simple and auditable

Batch execution is handled by separate orchestration layers
documented elsewhere.

Policy integration
------------------

Trained PPO policies are integrated using a dedicated **policy adapter**.

The adapter:

- Translates engine observations into PPO‑compatible inputs
- Converts PPO action outputs into engine‑legal actions
- Enforces a narrow interface between agent and engine

As a result:

- The engine remains agnostic to reinforcement learning algorithms
- The policy remains agnostic to engine internals
- Algorithm‑specific logic does not leak into execution code

- Acts strictly as an **execution bridge**
- Does not participate in opponent control, diagnostics or evaluation
- Produces a replay that is authoritative and engine‑native

Opponent control model
----------------------

During single‑match execution, **only one side of the match is controlled
by an external policy**.

The opposing side is controlled **internally by the environment**, using
a deterministic heuristic policy implemented as part of the engine.

This design:

- Eliminates co‑learning effects
- Reduces stochastic variance
- Improves interpretability of replays and diagnostics
- Allows observed behavior to be attributed to environment structure
  rather than opponent adaptation

Replay generation
-----------------

The replay produced by ``RLRunner`` is an **engine‑level replay** that
records structural game state transitions.

Key properties of the replay format:

- One snapshot per engine turn
- Actions recorded as explicit structural events
- End‑of‑turn and end‑of‑match markers emitted by the engine
- No dependency on PPO or reinforcement learning code

Replays can be:

- Visualized using the Python viewer
- Converted to event‑based web replays
- Inspected offline without loading models

Replay fidelity
---------------

The replay acts as a **ground‑truth record** of the executed match.

Because replays are generated by the engine itself:

- Game behavior can be validated visually
- Determinism can be verified independently
- Analysis can be repeated long after execution

This makes replays a central artifact for debugging,
validation and explanation.

Relation to offline analysis
----------------------------

The ``RLRunner`` is intentionally **not used** during batch evaluation
or offline analysis.

Instead:

- Batch execution runs matches directly via the engine session
- Diagnostics are collected through passive observers
- Outcomes are aggregated separately

This separation ensures that:

- Interactive execution remains simple
- Offline analysis remains independent and safe
- Responsibilities do not overlap across layers

Summary
-------

The ``RLRunner`` provides a minimal, deterministic and auditable way to:

- Execute a single RL‑controlled match
- Capture an authoritative replay
- Serve as a bridge between trained policies and analysis tools

It is a foundational component for reproducible experimentation,
but deliberately avoids any evaluative responsibility.
